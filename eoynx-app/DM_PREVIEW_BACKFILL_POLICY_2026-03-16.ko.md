# DM Preview 과거 데이터 정책 (2026-03-16)

## 결론
- 정책: 점진 갱신 채택
- 이유: E2E 암호화 구조상 서버에서 과거 메시지 본문을 복호화해 일괄 백필하기 어렵고, 대량 백필은 비용/리스크가 큼

## 적용 원칙
1. 신규 메시지부터 preview를 notifications.preview로 정상 적재
2. 과거 데이터는 강제 백필하지 않고, 사용자 상호작용/신규 트래픽으로 점진 치환
3. preview가 없는 과거 암호화 메시지는 기본 문구로 안전 fallback

## 구현 상태
- 웹 인박스 스레드 목록
  - encrypted 메시지 + preview 없음 -> "Encrypted message" 기본 문구 노출
- 모바일 인박스
  - 알림 preview 우선 사용, 없을 때 안전 fallback 유지

## 운영 가이드
- 단기
  - 사용자 문의 시: 과거 일부 스레드는 기본 문구가 보일 수 있음을 안내
- 중기
  - 트래픽이 많은 스레드부터 자연 치환 모니터링
- 장기
  - 필요 시 선택적 백필(최근 N일/활성 스레드만) 별도 배치 설계

## 리스크
- 과거 저활성 스레드는 기본 문구 상태가 길게 유지될 수 있음

## 후속 액션
- 모니터링 지표
  - 인박스에서 fallback 문구 비율
  - 신규 메시지 preview 적재 실패율
- 임계치 초과 시
  - 최근 활성 스레드 범위 백필 PoC 검토
